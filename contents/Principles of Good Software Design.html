Simplicity

“Make things as simple as possible, but no simpler.” –Albert Einstein


  • For whom and when
    • Is it simpler for you or for your clients?
    • Is it simpler for you to do now or maintain in the future?

Hide Complexity and Build Abstractions


Avoid Over-engineering

  • Beginning with a reasonable level of abstraction and iterating over it gives better results than trying to predict the future and build everything that might be needed later on.

Try Test-Driven Development

  • Test-driven development is a set of practices where engineers write tests first (Client’s point of view, how might they use it? What do they expected) and then implement the actual functionality.
    • No code without unit tests (no “spare” code)
    • A type of documentation (how the code was meant to be used and what the expected behavior was)

Loose Coupling

  • Coupling is a measure of how much two components know about

and depend on one another. (change in one part require change in other parts)


Promoting Loose Coupling

  • Carefully manage your dependencies.
    • When writing code, be stingy.
      • Share the absolute minimum
    • Java
      • Private, reduce coupling and promote simplicity
      • Public, reuse code (coupling)

Avoiding Unnecessary Coupling

  • Example of unnecessary coupling
    • Exposing all the private properties using public getters and setters
    • When clients of a module or class need to invoke methods in a particular order for the work to be done correctly.
    • Allowing circular dependencies
      • Draw dependency diagram to see your structure (should look like a tree rather than network graph)

Don’t Repeat Yourself (DRY)

  • Inefficient process
    • Continuous improvement with incremental changes
  • Lack of automation (deploying, compiling, testing, building …)
  • Reinventing the wheel
  • Copy/paste programming
    • Every change is likely need to be made in the copies
    • Get the team to commit to some rules “we never copt and paste code”
    • If reusing (not copy/paste) is the easiest way to get things done, everyone will use it. Otherwise, expected to see duplication or hacks.

Example of good practice

  • Validation of an e-mail
    • If the validation logic is put into the code that creates user account, you won’t be able to use it in a different context.

Draw Diagrams

  • Write high-level interface

  • Consider the most common use-case
    • Circuit-breaker design pattern

  • Sketch a draft of the interaction
  • Class Diagram


Single Responsibility

  • Summarize the responsibility of the class in a single sentence and put it in a comment on top of the class name.
    • “File Store is an application allowing clients to upload files and manage their metadata, and it also allows clients to find files based on complex searches.“

Open-Closed Principle

  • The open-closed principle is about creating code that does not have to be modified when requirements change or when new use cases arise.
    • A sorting algorithm that sorting only one class or A sorting algorithm that implements a comparator that can sort anything?

Dependency Injection

  • Dependency injection provides references to objects that the class depends on, instead of allowing the class to gather the dependencies itself.


Inversion of Control (IOC)

  • Inversion of control is not having to know who will create and use your objects, how, or when.

Design for Self-Healing

  • Highly-available in cases of crash (thousands of servers, crash happens on a daily bases)
  • Single point of failure is any piece of infrastructure that is necessary for the system to work properly.
  • Self-healing is the ability to detect and fix problems automatically without human intervention.
    • Cassandra (an open-source data store)
      • When a data node fails, Cassandra stops all in-coming traffic to the node until it is back. Because the remaining nodes in the cluster provide redundancy, the data can still be retrieved.
      • When blank node is added, it syncs over time by the system.
  • Mean time to recovery is one of the key components of the availability equation. The faster you can detect, react to, and repair, the higher your availability becomes.
    • Availability is actually measured as mean time to failure / (mean time to failure + mean time to recovery).
