Web Services as an Alternative Presentation Layer

  • Build the web application first and then add web services as an alternative interface to it
  • Hard to scale because everything is in one place


API-First Approach

  • Designing and building your API contract first and then building clients consuming that API and the actual implementation of the web service.
    • Maintain one piece of code that control the API
    • Easier to scale as API naturally divides into a set of smaller independent web services
  • To make sure you do not over-engineer and still provide all of the functionality needed by your clients, you may need to spend much more time designing and researching your future use cases


Pragmatic Approach

  • When you see a use case that can be easily isolated into a separate web service and that will most likely require multiple clients performing the same type of functionality, then you should consider building a web service for it.
  • On the other hand, when you are just testing the waters with very loosely defined requirements, you may be better off by starting small and learning quickly rather than investing too much upfront.

Types of Web Services

Function-Centric Services

  • Be able to call functions’ or objects’ methods on remote machines without the need to know how these functions or objects are implemented, in what languages are they written, or what architecture are they running on. (SOAP)
    • Everyone has to agree on how to pass parameters, given CPU architectures, and run-time environments.
    • Challenges: resource locking, security, network latencies, concurrency, and contracts upgrades.

Resource-Centric Services

  • There are only a few operations that can be performed on these objects (you can create, delete, update, and fetch them)
  • REST, very easy to use
    • Less secure, To allow authorized access to REST resources, web services usually require authentication to be performed before using the API.
    • The client would first authenticate (often using OAuth 2) and then provide the authentication token in HTTP headers of each consecutive request.
    • REST services also depend on transport layer security provided by HTTPS (HTTP over TLS Transport Layer Security) rather than implementing their own message encryption mechanisms.

Scaling REST Web Services

Keeping Service Machines Stateless

  • load balancing, add clones, crash-detecting, maintenance and update (no downtime), (potential auto-scaling if your cloud host supports it)


common use cases of sharing state between machines

  • Security, authentication token
    • That token will have to be validated on the web service side and make sure that the user has access to the operation they are attempting to perform.
    • Cache directly on web service machines
      • Problem, when changing permissions or blocking accounts, as these objects would need to expire before new permissions could take effect.
    • Shared in-memory object cache
      • Easily invalidate cache when users’ permissions change (cache is in one place rather than in thousands of machines)


Question whether transactions or even locking is necessary because they are so expensive to implement (Time, hurt scalability)
  • Resource locking
    • Distributed lock systems (Zookeeper) or even build your own simple lock service using a data store of your choice.
    • Avoid resource locks for as long as possible and look for alternative ways to synchronize parallel processes (optimistic concurrency control or message cues)
    • Locks require machines to talk to each other or to an external system. By using locks, all of your machines become interdependent.
      • If one process becomes slow, anyone else waiting for their locks becomes slow. (not desirable)
  • Application-level transactions
    • A distributed transaction is a set of internal service steps and external web service calls that either complete together or fail entirely.
      • 2 Phase Commit (2PC) algorithm
    • Increasingly difficult to perform as the number of services involved increases and more resources need to be available throughout the time of the transaction

  • Look for alternative
    • Compensating transaction, revert the result of an operation that was issued as part of a larger logical transaction that has failed. (roll back v.s. providing additional methods to revert)
    • Benefit, web services do not need to wait for one another; they do not need to maintain any state or resources for the duration of the overarching transaction either


Caching Service Responses

  • To take advantage of HTTP caching, you need to make sure that all of your GET method handlers are truly read-only. A GET request to any resource should not cause any state changes or data updates.
    • In some subtle ways, GET is not read-only.
      • If logs are created and needed, then GET would leave some changes.
      • Using local cache, since GET would update machine-level cache which would be different for each machine (state change)
    • Authentication, The URL is not enough to produce the response for the particular user. need to include the authentication headers when building the caching key.
      • This cache separation is good if your users should see different data, but it is wasteful if they should actually see the same thing.
  • To truly leverage HTTP caching, you want to make as many of your resources public as possible. Making resources public allows you to have a single cached object for each URL
  • Reverse Proxy, Load Balancing separated by service


Functional Partitioning

  • Functional partitioning is a way to split a service into a set of smaller, fairly independent web services, where each web service focuses on a subset of functionality of the overall system.
    • Few dependencies
    • Scale independently
      • Does it make sense to use the same caching for both of the services?
      • Does it make sense to use the same type of data store? Are both services equally critical to the business, and is the nature of the data they store the same?
      • Do you need to implement both of these vastly different web services using the same technology stack?
